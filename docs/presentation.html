Topology sets the shape. The policy sets the tradeoffs.

spec:
  # shape
  shards: 3            # 1 or >=3 (quorum)
  replicasPerShard: 2  # HA copies
  # capacity / deployment
  image: valkey/valkey:8
  storage:   { size: 8Gi }
  resources: { limits: { memory: 8Gi } }  # → maxmemory ~70%
  # durability ↔ throughput
  persistence: { mode: AOFAndRDB, appendFsync: everysec }
  # throughput ↔ memory
  performance: { ioThreads: 2, maxmemoryPolicy: noeviction }
  # availability ↔ consistency
  haPolicy: { minReplicasToWrite: 1, requireFullCoverage: true,
              clusterNodeTimeoutMillis: 5000 }
  • Topology — shards × replicasPerShard: capacity, write scale-out, HA copies.
  • resources — capacity: limits.memory sets maxmemory (~70%); image/storage are plumbing.
  • persistence — durability ↔ throughput: AOF / RDB / None, fsync cadence.
  • performance — throughput ↔ memory: io-threads, eviction policy.
  • haPolicy — availability ↔ consistency: write-safety gate, full-coverage, failover speed.
What actually matters

Two numbers are the minimum; the policy axes are the substance — real Valkey levers with real tradeoffs, each quantified in the benchmark (detailed next).

Slots, quorum, and asynchronous replication

  • 16384 hash slots; slot = CRC16(key) mod 16384. Each primary owns a range; together they cover all of them.
  • Failover is a vote among primaries → you need ≥3 (with 2 there's no majority). The CRD rejects shards: 2.
  • Replication is always asynchronous — no synchronous mode. An acked write can be lost if a primary dies before its replica sees it.
  • One role per node — a node is a master (owns slots) or a replica of a single master. Unlike MongoDB, it can't be primary for some slots and replica for others.
No synchronous replication

There is no sync mode in Redis/Valkey. min-replicas-to-write is a safety gate, not sync; WAIT is the per-write primitive. The data-loss window shrinks — it doesn't vanish.

The must-stay-up cache, at the node level

One StatefulSet per shard

  • One CRD — the whole API fits on a screen.
  • StatefulSet per shard: adding a shard = adding a StatefulSet; resharding has a clean unit.
  • Headless Service for stable per-pod DNS; announce-hostname + nodes.conf on the PVC so identity survives restarts.
  • Anti-affinity: a replica never shares a node with its primary.

The shape

Each shard = one StatefulSet of 1 + replicasPerShard pods (ordinals 0…N). One headless Service and one ConfigMap are shared across all shards of a cluster; everything is owner-referenced so deleting the CR garbage-collects it.

Observe → Decide → Act

  • Level-triggered: handed the object, reads full state, converges. No events to miss.
  • Idempotent: every step reads actual state first; safe to run a hundred times.
  • The decision is a pure function — no I/O, 100% unit-tested.
func Decide(desired, observed) Plan {
  if !Formed        → Form
  if !SlotsCovered  → Repair  // gate
  if want > have    → ScaleOut
  if want < have    → ScaleIn
  if replicasDiffer → ScaleReplicas
  else              → None
}

The choices that matter

  • One CRD, two numbers — no definition/instance/ops split to learn.
  • Pure Decide function — the hard question (what next) is I/O-free and testable; the mess (talking to a cluster) hides behind an interface.
  • Reuse Valkey's failover — quorum promotion already works; reimplementing it adds risk for nothing.
  • Native Go slot-mover for migration — deterministic where valkey-cli reshard refuses on uneven states.
  • Scope discipline — sharding/replication/resharding done right; documented non-goals stay out (YAGNI).

In one sentence

KubeBlocks is right if you run many databases. For one engine, a purpose-built operator you can read and test end-to-end in an afternoon — borrowing KubeBlocks' hard-won operational defaults: announce-hostname, nodes.conf on the PVC, drain-before-remove.

Why it stays correct under failure

The patterns

  • Read live state first — never assume identity, role, or count
  • Readiness gate before touching the cluster
  • Stability gate: repair open slots before any topology change
  • Bounded, resumable progress; idempotent external ops
  • Query every node where state isn't gossiped

Concrete cases handled

  • Operator crashes mid-reshard → repair + resume
  • Spec changed mid-reshard → converge to latest
  • Fast pod restart → resumes as primary, no needless failover
  • Teardown driven by existing StatefulSets, not primary count

Resharding moves slots, not keys

  • A key's slot is fixed: slot = CRC16(key) mod 16384, and routing is purely by slot → owner.
  • So you can't relocate data by moving keys — clients would still route to the slot's owner. You transfer the slot; its keys come along.
  • Resharding = rewriting slot → primary ownership to rebalance when shards are added or removed.
  • Data-preserving: a key requested mid-move is redirected to the new owner.
Move slots, not keys

You're rewriting the routing map; the keys move as a side effect. It's data-preserving and converges from any starting distribution.

One framework, layered by what each test needs to run

  • Unit — pure logic, nothing to start: slot math, the Decide planner, config render (topology 100%).
  • envtest — the reconciler against a real API server with the fake ClusterAdmin — no live Valkey, no Kind.
  • kind e2e — the three user stories as one ordered run: provision, failover, reshard 3→5, replica scale, scale-in 5→3.
  • bench — separate, and not Go: valkey-benchmark driving the live cluster.
All Go's std testing

No BDD framework — every test and subtest runs and debugs natively in the IDE, and go test ./... is the whole suite.

Choices that matter

  • The seam — a ClusterAdmin interface with an in-memory fake makes the reconciler testable without a live Valkey.
  • Self-provisioning — envtest boots its own API server; e2e deploys the operator + CR and tears down just the instance.
  • Safe by default — e2e refuses any non-Kind context, so it can never run against a real cluster.

~225s on Kind — all five day-2 flows green against a real Valkey cluster.

The 60-second version

A Kubernetes operator for Valkey Cluster. You declare shards and replicas-per-shard; it reconciles a real, sharded, HA cluster — one StatefulSet per shard, hostname identity that survives restarts, data-preserving resharding when you change the topology. A pure Decide function picks one action; a ClusterAdmin interface carries it out. The interesting tradeoffs are exposed as knobs and quantified in a benchmark. Built spec-first, cross-checked against KubeBlocks, and hardened by stressing it until it broke — then fixing the root cause.
