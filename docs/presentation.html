The approach

Small scope, done correctly and understood deeply — every requirement traceable to code and docs.

You declare two numbers

apiVersion: cache.razkevich.dev/v1alpha1
kind: ValkeyCluster
spec:
  shards: 3            # 1 or >=3 (quorum)
  replicasPerShard: 2  # HA copies
  persistence: { mode: AOFAndRDB }
  performance: { ioThreads: 2 }
  haPolicy:
    minReplicasToWrite: 1
    requireFullCoverage: true
    clusterNodeTimeoutMillis: 5000
  • Topology is two numbers; everything else is optional tuning on three axes.
  • Status is the truth, derived from the live cluster every reconcile.
  • Monitoring surface is kubectl — no extra stack.

One StatefulSet per shard

  • One CRD — the whole API fits on a screen.
  • StatefulSet per shard: adding a shard = adding a StatefulSet; resharding has a clean unit.
  • Headless Service for stable per-pod DNS; announce-hostname + nodes.conf on the PVC so identity survives restarts.
  • Anti-affinity: a replica never shares a node with its primary.

Why it matters

A shard maps to exactly one workload — that single decision makes every day-2 operation legible.

Observe → Decide → Act

  • Level-triggered: handed the object, reads full state, converges. No events to miss.
  • Idempotent: every step reads actual state first; safe to run a hundred times.
  • The decision is a pure function — no I/O, 100% unit-tested.
func Decide(desired, observed) Plan {
  if !Formed        → Form
  if !SlotsCovered  → Repair  // gate
  if want > have    → ScaleOut
  if want < have    → ScaleIn
  if replicasDiffer → ScaleReplicas
  else              → None
}
The testability seam

A ClusterAdmin interface carries actions out — go-redis + pod-exec in prod, an in-memory fake in tests — so the reconciler runs in envtest with no live Valkey.

The choices that matter

  • One CRD, two numbers — no definition/instance/ops split to learn.
  • Pure Decide function — the hard question (what next) is I/O-free and testable; the mess (talking to a cluster) hides behind an interface.
  • Reuse Valkey's failover — quorum promotion already works; reimplementing it adds risk for nothing.
  • Native Go slot-mover for migration — deterministic where valkey-cli reshard refuses on uneven states.
  • Scope discipline — sharding/replication/resharding done right; documented non-goals stay out (YAGNI).

In one sentence

KubeBlocks is right if you run many databases. For one engine, a purpose-built operator you can read and test end-to-end in an afternoon — borrowing KubeBlocks' hard-won operational defaults: announce-hostname, nodes.conf on the PVC, drain-before-remove.

Slots, quorum, and an honest data-loss window

  • 16384 hash slots; slot = CRC16(key) mod 16384. Each primary owns a range; together they cover all of them.
  • Failover is a vote among primaries → you need ≥3 (with 2 there's no majority). The CRD rejects shards: 2.
  • Replication is always asynchronous — no synchronous mode. An acked write can be lost if a primary dies before its replica sees it.
No synchronous replication

There is no sync mode in Redis/Valkey. min-replicas-to-write is a safety gate, not sync; WAIT is the per-write primitive. The data-loss window shrinks — it doesn't vanish.

Quantified

Benchmarks measure each: throughput vs shard count · WAIT durability cost (3.3s→5.4s) · failover window vs node-timeout (5s→~1.6s) · io-threads sweep.

What I'd reach for next (and know cold)

  • min-replicas-max-lag (10s) — the lag bound that companions min-replicas-to-write; the gate is meaningless without it.
  • cluster-replica-validity-factor — may a stale replica promote? availability vs data-recency.
  • cluster-migration-barrier / allow-replica-migration — auto-cover an orphaned primary.
Deliberate scope

The headline failover/durability levers are exposed; these second-order ones run at Valkey defaults — and the first two are the candidates to promote to CRD fields to make the HA story airtight.

Drop durability, buy availability, spend on speed

The CR

  • persistence: None — no fork, no fsync, no COW spike → push maxmemory higher
  • requireFullCoverage: false — serve the slots that are up (the headline lever)
  • minReplicasToWrite: 0 — don't block writes for durability you don't want
  • maxmemoryPolicy: allkeys-lfu — it's a cache; evict

OS / node level

  • Transparent Huge Pages off — the #1 latency fix
  • Swap off / swappiness 0
  • somaxconn + tcp-backlog raised together; high fd limit
  • Guaranteed QoS + PodDisruptionBudget so node drains keep quorum

Why it stays correct under failure

The patterns

  • Read live state first — never assume identity, role, or count
  • Readiness gate before touching the cluster
  • Stability gate: repair open slots before any topology change
  • Bounded, resumable progress; idempotent external ops
  • Query every node where state isn't gossiped

Concrete cases handled

  • Operator crashes mid-reshard → repair + resume
  • Spec changed mid-reshard → converge to latest
  • Fast pod restart → resumes as primary, no needless failover
  • Teardown driven by existing StatefulSets, not primary count

The things you only learn by running it

  • CLUSTER MEET needs an IP, not a hostname — so resolve FQDN→IP for MEET while the node still announces its hostname.
  • Open-slot markers live only on the owning node's own line — they're not gossiped, so detecting a stuck migration means querying every node.
  • SETSLOT is rejected on replicas — assert ownership on masters only.
  • Primary ≠ pod ordinal 0 after a failover — discover it via CLUSTER MYID + slot ownership.
Why it matters

This is where running a clustering operator differs from reading about one — and where the unit tests were green but the cluster wasn't.

How they were found

Each surfaced by deliberately stressing the operator — multiple clusters, mid-reshard topology flips — then fixed at the root cause and pinned with a regression test. Changing the desired topology mid-reshard now converges with zero data loss.

The division of labor

AI gave leverage on scaffolding and boilerplate; human judgment owned the correctness-critical parts — the consistency tradeoffs, migrate-before-remove ordering, and the gotchas that only surfaced live.

Deliberately bounded

In scope: sharding, replication, data-preserving resharding, failover, truthful status. Out by design: rolling upgrades, vertical scaling, volume expansion, backup/restore, TLS/ACL, Sentinel (cluster-mode failover covers HA; shards: 1 is the HA-only case), a proxy, and a metrics stack.

The principle

A smaller surface done correctly and understood deeply beats an ambitious half-built operator — and it's the honest answer to the brief.

The 60-second version

A Kubernetes operator for Valkey Cluster. You declare shards and replicas-per-shard; it reconciles a real, sharded, HA cluster — one StatefulSet per shard, hostname identity that survives restarts, data-preserving resharding when you change the topology. A pure Decide function picks one action; a ClusterAdmin interface carries it out. The interesting tradeoffs are exposed as knobs and quantified in a benchmark. Built spec-first, cross-checked against KubeBlocks, and hardened by stressing it until it broke — then fixing the root cause.
